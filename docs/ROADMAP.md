# Aurexis — Roadmap (phased)

Each phase is **shippable** and has an explicit **acceptance test**. Product work
is additive; the faithful prototype (`v2.1-react-screens`) stays the visual
reference until a screen is deliberately wired to the backend. "Stays simulated"
= still uses the prototype's hardcoded/illustrative behaviour until a later phase
replaces it.

---

## Phase 0 — Foundation + plumbing slice  *(this phase)*

**Scope.** Planning docs (`ARCHITECTURE.md`, `ROADMAP.md`, `SETUP.md`); backend
skeleton (`/api/health`, `/api/ai-ping`, `/api/db-ping`, `_lib/anthropic`,
`_lib/supabase`); initial Supabase migration (authored, not required to run);
env/secret scaffolding (`.env.example`, gitignore); a hidden `/dev-check` page
that proves frontend → function → Claude and frontend → function → Supabase.

**Dependencies.** Node toolchain; (for the human to run the slice) a Supabase
project + an Anthropic API key.

**Acceptance test.**
- `npm install` (root) succeeds; `/api/health` returns `{ok:true, ts}`.
- With real keys in `.env.local` and `vercel dev` running, the `/dev-check` page:
  "Ping AI" shows a Claude reply; "Ping DB" shows `ok` (or a clean error if the
  migration isn't applied).
- No prototype screen changed; no secret committed.

**Stays simulated.** Everything in the prototype — all engines still use
hardcoded data; the backend is not wired to any prototype screen.

---

## Phase 1 — Auth (Supabase) + persistence

**Scope.** Supabase Auth in the React app (email magic-link / password, anon key
in browser). Sign up / sign in / sign out. Apply the `0001_init` migration. Save
a user's profile + session and **resume** it on next sign-in. A small account
menu; the prototype flow runs for signed-in users and persists answers.

**Dependencies.** Phase 0 skeleton; migration applied; `VITE_SUPABASE_*` set.

**Acceptance test.** A new user signs up, completes Profile, reloads / signs out
and back in, and their answers + current stage are restored from
`public.sessions`. RLS verified: user A cannot read user B's rows (direct query
with A's token returns nothing).

**Stays simulated.** Opportunity/blueprint/forge/sentinel/studio content is still
the prototype's hardcoded data; only auth + persistence are real.

---

## Phase 2 — Real AI for Profile → Discover → Blueprint

**Scope.** Replace hardcoded `LIB` / question scoring / blueprint strings with
**server-side Claude** generations behind `/api/profile`, `/api/discover`,
`/api/blueprint`. Each validates Claude's output against a JSON schema before
returning. Persist generations to `sessions`. UI renders the same components,
now fed by real data.

**Dependencies.** Phases 0–1; `ANTHROPIC_API_KEY` set; schemas defined.

**Acceptance test.** For two different profiles, Discover returns **distinct,
relevant, schema-valid** opportunities generated by Claude (not from `LIB`);
Blueprint reflects the chosen opportunity; results persist and reload identically.
Latency and token spend logged.

**Stays simulated.** Forge/Sentinel/Studio remain prototype animations.

---

## Phase 3 — Forge: real generation (architecture + scaffold)

**Scope.** `/api/forge` asks Claude for an architecture + a **starter project
scaffold** (a set of files) for the chosen blueprint. Output is assembled into a
downloadable project (zip / writes to a repo). Human-in-the-loop: the user
reviews and chooses what to keep. **No autonomous deploy.**

**Dependencies.** Phase 2 blueprint output; a file-assembly/zip step; storage for
generated artifacts.

**Acceptance test.** Forge produces a real, inspectable scaffold (e.g. a runnable
starter that builds) for a chosen blueprint; the user can download it; copy
describes it as a scaffold, not a shipped business.

**Stays simulated.** Sentinel/Studio still illustrative.

---

## Phase 4 — Sentinel: real audit on Forge output

**Scope.** `/api/sentinel` runs a real audit over the Forge scaffold:
**dependency scan** (npm audit / OSV), **static analysis** (lint + security
rules), and an **AI review against a fixed rubric**. Returns a findings report
with severities. Copy: "audited / issues flagged", never "hack-proof".

**Dependencies.** Phase 3 artifacts; scanner integration; rubric defined.

**Acceptance test.** Given a scaffold with a known vulnerable dependency and an
intentional insecure pattern, Sentinel flags both with severity and location; a
clean scaffold reports "no issues found by these checks" (not "secure").

**Stays simulated.** Studio still illustrative.

---

## Phase 5 — Studio: real GEO measurement + recommendations

**Scope.** `/api/studio` measures how the product is surfaced across AI engines
(query real models/search via API), reports a baseline discoverability signal,
and returns concrete content/structure recommendations. Copy: "improves
discoverability", "measured presence" — never a ranking guarantee.

**Dependencies.** A live product/landing to measure; engine query APIs; baseline
metric defined.

**Acceptance test.** For a real target, Studio returns a measured baseline across
≥2 engines and actionable recommendations; re-measuring after applying a
recommendation shows the metric is tracked over time. No guarantee language.

**Stays simulated.** Nothing — all five engines are real after this phase.

---

## Phase 6 — Polish, limits, cost, (optional) billing

**Scope.** Rate limiting + abuse protection on `/api`; robust error handling and
user-facing error states; per-user **usage + cost tracking** for Claude calls;
caching where safe; observability/logging; optional billing (Stripe) and plan
limits.

**Dependencies.** Phases 1–5.

**Acceptance test.** Abusive call patterns are throttled with a clear error;
every AI call records tokens + estimated cost per user; a forced failure surfaces
a friendly message, not a stack trace; (if billing) plan limits are enforced.

**Stays simulated.** Nothing.
