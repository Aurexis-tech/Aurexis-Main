# Aurexis — Architecture (Phase 0 foundation)

> Status: **foundation only.** This document defines the buildable system. No
> product features are wired into the prototype UI yet. The faithful prototype
> (static baseline + the React port, tag `v2.1-react-screens`) is the preserved
> visual reference; all product work is **additive** on top of it.

## 1. The stack (decided) and why

| Layer | Choice | Why |
|-------|--------|-----|
| Frontend | **Vite + React** (the existing app under `./react/`, kept as-is) | Already built and parity-verified; no rewrite. Static, cheap to host, fast. |
| API | **Vercel serverless functions** (`/api/*` at repo root) | Zero-ops HTTP endpoints that sit next to the static frontend on the same platform. Secrets live here, never in the browser. Scales to zero. |
| DB + Auth | **Supabase (Postgres + Auth)** | Managed Postgres with Row-Level Security and a batteries-included auth system (email/OAuth). RLS lets us enforce per-user data isolation in the database itself. |
| AI | **Anthropic Claude** (`@anthropic-ai/sdk`, model `claude-sonnet-4-6`) | All "intelligence" in the engines (profiles, opportunities, blueprints, code, audits, GEO copy) is generated by Claude **server-side**. |

**Trust boundary.** The browser is untrusted. The `ANTHROPIC_API_KEY` and the
Supabase **service-role** key are used **only inside `/api` functions** and are
never bundled into client code. The browser may hold the Supabase **anon** key
(safe by design — it is gated by RLS) once Phase 1 adds client auth.

```
 Browser (Vite/React)  ──fetch──▶  /api/* (Vercel function)  ──▶  Claude  (ANTHROPIC_API_KEY)
   anon key only                     service-role key       ──▶  Supabase (SUPABASE_SERVICE_ROLE_KEY)
```

## 2. Honest scope translation (what each engine REALLY is)

The prototype's on-screen copy is aspirational. These are the **real, buildable**
definitions we will implement — and the copy we will actually ship.

- **Profile / Discover / Blueprint — real Claude-generated content.**
  Replace the hardcoded `LIB` / question tables / blueprint strings with live
  Claude generations (server-side). The user's answers go to a function; Claude
  returns a structured profile, a ranked set of *real* opportunities for their
  domain, and a concrete blueprint. Output is validated against a JSON schema
  before it reaches the UI.

- **Forge — real AI architecture + scaffold generation, human-in-the-loop.**
  Claude produces an architecture and **starter code / a project scaffold** that
  is emitted as a real, downloadable project (files in a repo/zip), plus a build
  plan. It is **not** an autonomous agent that ships arbitrary software to
  production. A human reviews, edits and decides what to run. Copy: "generates a
  starter project / scaffold", never "builds and launches a business by itself".

- **Sentinel — real automated security audit.**
  Runs concrete checks on Forge output: **dependency vulnerability scan**
  (e.g. `npm audit` / OSV), **static analysis** (lint/security rules), and an
  **AI review against a fixed rubric**. It produces a report of findings with
  severities. Copy says **"audited"**, **"issues flagged"**, "hardened against the
  checked categories" — **never "hack-proof"** or "unhackable". A clean run means
  "no issues found by these checks", not "secure".

- **Studio — real GEO measurement + recommendations.**
  Measures how the product is currently surfaced across AI engines (query real
  models/search via API), reports a baseline discoverability signal, and returns
  **content/structure recommendations** to improve it. Copy: **"improves
  discoverability"**, "measured presence across engines" — **never "guaranteed
  #1"** or any ranking guarantee.

### Copy rule (enforced)

> Product copy must **not** contain false guarantees. Banned claims include
> "hack-proof", "unhackable", "100% secure", "guaranteed #1", "guaranteed
> ranking", "autonomously launches a business". Use measured, hedged language:
> *audited, issues flagged, improves, estimated, projected (illustrative)*. The
> prototype's illustrative figures must be labelled as such wherever shown.

## 3. Data model overview

Authoritative schema lives in `supabase/migrations/0001_init.sql` (authored;
applied by a human via the Supabase CLI/dashboard). Phase 0 tables:

- **`public.health`** — a 1-row table used only by `/api/db-ping` to prove
  connectivity. No user data.
- **`public.profiles`** — one row per `auth.users` id. Holds the user's answer
  signature (`answers` jsonb), a derived `profile_label`, timestamps. RLS:
  **owner-only** read/insert/update.
- **`public.sessions`** — a saved run of the engine for resume. Columns: `stage`
  (current screen), `answers`, `chosen` (opportunity), `blueprint`, free-form
  `state`, timestamps, `user_id → auth.users`. RLS: **owner-only** CRUD.

Every user-owned table has **Row-Level Security enabled** with `auth.uid()`-based
policies, so the database — not just the API — enforces isolation. The
service-role key (server functions) bypasses RLS intentionally and must stay
server-side.

## 4. API surface

Phase 0 (plumbing only — none wired into prototype screens):

| Endpoint | Method | Purpose | Secrets used |
|----------|--------|---------|--------------|
| `/api/health` | GET | Liveness probe `{ok, ts}` | none |
| `/api/ai-ping` | GET | Calls Claude with a fixed trivial prompt, returns the text | `ANTHROPIC_API_KEY` (server) |
| `/api/db-ping` | GET | Trivial Supabase connectivity check, returns ok/error (no key leakage) | `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY` (server) |

Shared server libs (never imported by the browser):

- `/api/_lib/anthropic.js` — Claude client + `complete()` helper, reads
  `ANTHROPIC_API_KEY`, pins model `claude-sonnet-4-6`.
- `/api/_lib/supabase.js` — admin Supabase client from `SUPABASE_URL` +
  `SUPABASE_SERVICE_ROLE_KEY`.

Future endpoints (later phases, sketched): `/api/profile`, `/api/discover`,
`/api/blueprint`, `/api/forge`, `/api/sentinel`, `/api/studio` — each a
server-side Claude call returning schema-validated JSON, gated by the caller's
Supabase auth token.

## 5. Env / secret strategy

- All secrets come from **environment variables** — never hardcoded, never
  committed. `.env.example` lists **names only** (no values).
- `.env` and `.env.local` are git-ignored; the human supplies real values
  locally (see `docs/SETUP.md`).
- **Server-only** (used by `/api`, loaded by `vercel dev` from repo-root
  `.env.local`): `ANTHROPIC_API_KEY`, `SUPABASE_URL`,
  `SUPABASE_SERVICE_ROLE_KEY`.
- **Browser-safe** (Phase 1, Vite-exposed from `react/.env.local`, must be
  `VITE_`-prefixed): `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`. The anon key
  is safe in the browser **only because RLS restricts it**; the service-role key
  is **never** exposed to the browser.
- In production these are set as Vercel project Environment Variables, not files.

## 6. What is NOT in Phase 0

No prototype screen calls the backend yet. The only runnable proof is the hidden
`/dev-check` page (frontend → function → Claude, and → Supabase). Real engine
wiring, auth, and persistence arrive in Phases 1–5 (`docs/ROADMAP.md`).
